01 ¿Qué es?

En simple: el sistema que coordina una cadena de tiendas. Maneja ventas en caja, inventario por sucursal, control de stock, conciliación financiera y cumplimiento tributario chileno (boletas, facturas, notas de crédito).

En técnico: Sistema transaccional con concurrencia controlada por locks pesimistas (SELECT FOR UPDATE), atomicidad multi-entidad vía Unit of Work, RBAC granular basado en perfiles, audit trail inmutable, rotación de refresh tokens con detección de replay, y FEFO (First Expired, First Out) automático en productos perecibles.

05 Decisiones técnicas notables

Concurrencia con locks pesimistas

SELECT FOR UPDATE sobre stock, folios SII y sesiones de caja. Evita race conditions reales en venta concurrente sin depender de retries optimistas.

Unit of Work por caso de uso

Cada use case abre/cierra su propio UoW. Cualquier excepción revierte todo el flujo. La auditoría se publica dentro de la misma transacción (consistencia fuerte).

Rotación de refresh tokens con detección de replay

Al usar un refresh, se emite uno nuevo y el anterior queda revocado. Si alguien intenta reutilizar uno ya revocado, se dispara ERR_REFRESH_REVOCADO y se invalidan TODOS los tokens de ese usuario.

Asignador de folios SII centralizado

Lock pesimista sobre RangoFolios controla agotamiento y permite gestión de rangos por sucursal y tipo de documento sin conflictos entre cajas.

Cálculo IVA con redondeo back-out

iva = round(bruto * 19 / 119) para precios con IVA incluido. Es la fórmula correcta del SII y evita el error clásico de calcularlo hacia adelante y perder un peso.

RBAC dinámico, no hardcoded

Perfiles configurables desde el módulo de Administración. Verificación de permisos en cada caso de uso (defense in depth), no solo en UI. Decorador @requires_permission.

Modelos ORM separados del dominio

Entidades de dominio son @dataclass puros sin dependencia de SQLAlchemy. Mappers explícitos en repositorios. Permite testear el dominio sin DB.

Cliente HTTP frontend con interceptor single-flight

Refresh automático al 401, single-flight para evitar tormentas de refresh concurrentes, y anti-loop para detectar refresh que también falla.

06 Glosario de patrones

Clean Architecture

Diseño en capas (dominio → aplicación → infraestructura) donde las dependencias siempre apuntan hacia el dominio. La lógica de negocio no conoce la base de datos ni el framework, así que se puede testear y cambiar la infraestructura sin tocarla.

SOLID

Cinco principios de diseño orientado a objetos: responsabilidad única, abierto/cerrado, sustitución de Liskov, segregación de interfaces e inversión de dependencias. Guían hacia código desacoplado y mantenible.

Unit of Work (UoW)

Patrón que agrupa todas las operaciones de base de datos de una operación de negocio en una sola transacción: o se confirman todas (commit) o no se aplica ninguna (rollback). Garantiza atomicidad: nunca queda media venta registrada.

Repository pattern

Abstrae el acceso a datos detrás de una interfaz. El dominio pide "guardar" o "buscar" sin saber si detrás hay PostgreSQL, otra BD o memoria. Permite testear el dominio sin base de datos real.

RBAC (Role-Based Access Control)

Control de acceso basado en roles: los permisos se asignan a perfiles y los usuarios heredan los permisos de su perfil, en vez de asignarlos uno a uno. "Granular" significa que el permiso es a nivel de acción concreta (recurso.accion), no de pantallas completas.

Lock pesimista vs. optimista

El lock pesimista (SELECT FOR UPDATE) bloquea la fila en la BD antes de modificarla, asumiendo que habrá conflicto; otra transacción debe esperar. El optimista no bloquea: deja avanzar y reintenta si detecta que el dato cambió. Usé pesimista en stock, folios y caja porque el conflicto es real y caro.

Audit trail

Registro inmutable (solo se agrega, nunca se edita ni borra) de quién hizo qué, cuándo y con qué resultado, guardando el estado antes/después. Sirve para trazabilidad, depuración y cumplimiento.

FEFO (First Expired, First Out)

"Primero en vencer, primero en salir": al vender, el sistema descuenta primero el lote con fecha de vencimiento más próxima. Reduce las pérdidas por caducidad en productos perecibles.

JWT RS256

JSON Web Token firmado con RSA + SHA-256 (criptografía asimétrica): el servidor firma con clave privada y cualquier servicio valida con la clave pública. Más seguro que HS256 (clave compartida) cuando varios servicios verifican el token.

Argon2id

Algoritmo moderno de hashing de contraseñas, ganador del Password Hashing Competition. Es deliberadamente costoso en memoria y CPU para resistir ataques de fuerza bruta con GPU; la variante id combina defensa contra GPU y side-channels.

Rotación de refresh tokens + detección de replay

Cada vez que se usa un refresh token se emite uno nuevo y el anterior se revoca. Si llega uno ya revocado se asume robo (replay) y se invalidan todas las sesiones de ese usuario. "Replay" = reutilizar un token/credencial que ya fue consumido.

Single-flight

Cuando varias peticiones disparan el mismo refresh al mismo tiempo, se ejecuta una sola y las demás esperan ese resultado. Evita la "tormenta" de múltiples refresh concurrentes tras un 401.

Defense in depth

Seguridad en capas: el permiso se valida en el backend (en cada caso de uso), no solo ocultando el botón en la UI. Si una capa falla, otra sigue protegiendo.

IVA back-out (redondeo)

Para precios "IVA incluido", el IVA se extrae desde el bruto con iva = round(bruto · 19 / 119) en lugar de calcularlo hacia adelante. Es la fórmula correcta del SII y evita el descuadre clásico de $1.

¿Por qué lo construyo?

Lo construyo para practicar lo que no uso a diario: arquitectura limpia desde cero, pruebas serias, concurrencia y seguridad aplicada. Hay decisiones técnicas, como los locks pesimistas, la atomicidad multi-entidad o el RBAC granular, que solo se entienden a fondo cuando las implementas tú mismo.
